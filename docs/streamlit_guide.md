# Streamlit 화면 구성

---

## 1페이지 — 프로젝트 개요

프로젝트 전체 맥락과 핵심 결과를 한눈에 파악할 수 있는 요약 페이지

### 주요 구성

**KPI 카드 (4개)**
| 항목 | 값 |
|---|---|
| 분석 고객 수 | 500명 (account 단위) |
| 입력 피처 수 | 74개 (전처리 후 기준) |
| Churn 비율 | 22% (불균형 분류 문제) |
| 최적 Threshold | 0.45 · Logistic Regression · F1 0.433 |

**카드 섹션 (2×2)**
- **문제 정의**: SaaS 환경에서 이탈을 사후 확인하는 것이 아닌 선제 탐지와 개입이 목표임을 설명
- **핵심 결과 요약**: 최적 모델, 운영 Threshold, 대표 이탈 신호를 태그 형태로 요약
- **모델 운영 기준**: accuracy가 아닌 recall·F1을 기준으로 삼은 이유와 threshold tuning 필요성 설명
- **분석 흐름**: EDA → 모델 비교 → XAI → 유지 전략으로 이어지는 4단계 순서 안내

**핵심 질문 3개**
> 어떤 고객이 이탈 위험이 높은가?
> 모델은 무엇을 근거로 판단했는가?
> 그 고객을 유지하기 위해 무엇을 할 수 있는가?

## 2페이지 — EDA

이탈 고객과 유지 고객의 특성 차이를 비교해 이탈 신호를 탐색하는 페이지

### 주요 구성

**Churn 여부별 평균 차이 테이블**

47개 수치형 변수에 대해 churn 고객 평균, non-churn 고객 평균, 차이값(diff)을 계산해 차이가 큰 순으로 정렬

- 상위 변수 예시: `total_usage_duration_secs` (churn 고객이 오히려 높음), `avg_mrr_amount` (churn 고객이 약 232 낮음), `days_since_last_usage` (churn 고객이 약 5일 낮음)
- 단순 수치 차이를 통해 이후 모델링과 XAI 해석의 출발점으로 활용

**시각화 4개**

| 그래프 | 내용 |
|---|---|
| 전체 Churn 분포 | 이탈(22%) vs 유지(78%) 비율 — 클래스 불균형 확인 |
| 핵심 변수 상관관계 히트맵 | 21개 주요 변수 간 상관관계 — 유사 변수 묶음 파악 |
| 사용량 평균 비교 | churn 여부별 `total_usage_count` 평균 차이 |
| Health Score 평균 비교 | churn 여부별 `health_score` 평균 차이 |

각 시각화 아래에 짧은 해석 note를 제공해 처음 보는 사람도 맥락을 이해할 수 있게 구성

## 3페이지 — 모델 성능

어떤 모델이 좋은지보다, 어떤 기준으로 운영할지를 결정하는 단계

### 주요 구성

**KPI 카드 (3개)**
| 항목 | 값 |
|---|---|
| 기본 성능 기준 모델 | Logistic Regression (ROC-AUC 0.612) |
| 최적 Threshold | 0.45 |
| 최고 F1 | 0.433 |

**탭 1 — 기본 비교 (threshold 0.5)**

threshold 0.5 고정 시 세 모델의 성능 비교 테이블

| 모델 | Accuracy | Precision | Recall | F1 | ROC-AUC |
|---|---|---|---|---|---|
| Logistic Regression | 0.240 | 0.197 | 0.765 | 0.313 | 0.612 |
| Random Forest | 0.227 | 0.227 | 1.000 | 0.370 | 0.398 |
| DL MLP | 0.787 | 0.000 | 0.000 | 0.000 | 0.432 |

> DL MLP는 threshold 0.5에서 예측을 아예 못 하는 상황이 발생

**탭 2 — Tuned 비교**

각 모델에 최적 threshold를 적용한 뒤 성능 비교 테이블

| 모델 | Threshold | Recall | F1 |
|---|---|---|---|
| Logistic Regression | 0.45 | 0.813 | 0.433 |
| Random Forest | 0.20 | 0.875 | 0.364 |
| DL MLP | 0.15 | 1.000 | 0.356 |

**탭 3 — Threshold 곡선**

threshold 0.05~0.95 구간에서 F1 / Precision / Recall 변화를 세 모델 모두 겹쳐 line chart로 시각화
- Logistic Regression: 0.45 부근에서 F1 정점
- Random Forest·DL MLP: 낮은 threshold에서 빠르게 0으로 수렴

**실무 해석 카드 (2개)**
- **왜 Threshold Tuning이 필요한가**: recall과 precision 균형을 업무 목적에 맞게 조정하는 이유 설명
- **결론**: Logistic Regression · threshold 0.45 · F1 0.433이 가장 균형 잡힌 운영 기준임을 제시

## 4페이지 — 이탈 요인 & 유지 전략

SHAP 기반으로 이탈 신호를 해석하고 실행 가능한 고객 유지 전략을 제안하는 페이지

### 주요 구성

**대표 이탈 신호 카드 (3개)**
| 순위 | 변수 | 한글명 |
|---|---|---|
| #1 | `active_subscription_ratio` | 활성 구독 비율 |
| #2 | `error_rate` | 오류 발생 비율 |
| #3 | `industry_DevTools` | DevTools 산업군 |

**탭 1 — XAI 요약표**

SHAP mean_abs_shap 기준 상위 15개 변수를 5개 컬럼으로 구성된 테이블로 제공

- `feature (원문)` — 모델 학습에 사용된 원본 변수명
- `한글 변수명` — 이해하기 쉽게 풀어쓴 이름
- `중요도` — mean_abs_shap 값 (클수록 예측에 더 큰 영향)
- `설명` — 변수가 실제로 무엇을 의미하는지
- `해석 포인트` — 값이 높거나 낮을 때 어떤 의미인지

하단에 "처음 보시는 분을 위한 읽는 법" expander를 제공

**탭 2 — SHAP 시각화**

| 그래프 | 내용 |
|---|---|
| Summary Plot (beeswarm) | 각 변수가 churn 확률을 높이는 방향인지 낮추는 방향인지 색상으로 표시 |
| Bar Plot | 전체 고객 기준 평균 영향력 상위 변수 순위 |

**탭 3 — 유지 전략**

| 전략 | 대상 | 액션 |
|---|---|---|
| 전략 1. 사용량 저하 고객 선제 케어 | 사용량 감소·last_usage 증가·활성 구독 비율 저하 고객 | 튜토리얼, 리마인드 메일, 기능 재활성화 캠페인 |
| 전략 2. 오류 경험 고객 즉시 대응 | error_rate 상승·응답 지연 고객 | 우선 대응 SLA, 전담 지원, 장애 후속 안내 |
| 전략 3. 고위험 고객군 세분화 운영 | 전체 고위험 고객 | 사용 저하형·장애 불만형·응답 지연형으로 유형화해 정교한 retention 설계 |

## 5페이지 — 고객별 예측

개별 고객의 이탈 위험도를 확인하고 실제 이탈 여부와 비교하는 페이지

### 주요 구성

**고객 선택**

테스트셋 75명의 account_id 드롭다운에서 분석할 고객을 선택

**예측 결과 (3개 metric)**

| 항목 | 내용 |
|---|---|
| Logistic Regression | 예측 확률 + 고위험 / 안정 태그 + threshold 표시 |
| DL MLP | 예측 확률 + 고위험 / 안정 태그 + threshold 표시 |
| 실제 churn_flag | 0 — 유지 / 1 — 이탈 |

**추천 액션 카드**

Logistic Regression 기준으로 예측 확률과 threshold 거리에 따라 3단계 메시지를 자동 출력

| 조건 | 메시지 |
|---|---|
| 확률 ≥ threshold + 0.15 | 즉시 케어 대상 — 전담 CS 연결, 사용 저해 요인 확인, 핵심 기능 재안내 |
| 확률 ≥ threshold | 주의 고객 — 리텐션 메시지 발송, 사용량 추적, 이슈 여부 점검 |
| 확률 < threshold | 안정 고객 — 사용량 감소·오류율 상승 여부 지속 모니터링 |

**하단 2컬럼**
- **왼쪽**: 선택 고객의 주요 feature 15개 데이터프레임 미리보기
- **오른쪽**: 해석 포인트 카드 — 테스트셋 기준 분석임을 안내

**전체 테스트셋 예측 비교 테이블**

테스트셋 75명 전원의 Logistic Regression 예측 확률, DL MLP 예측 확률, 실제 churn_flag를 한 테이블에서 비교할 수 있음.
